# 개선 제안 (evolve)

이번 작업에서 **반복된 문제·마찰**을 모아, 규칙·문서·플러그인을 어떻게 고치면 좋을지 **제안**으로 정리한다. 한 바퀴(seed→build→evaluate→review)의 마지막 단계이며, 다음 작업이 더 나아지게 하는 자기개선 고리다.

## 목적

- 같은 문제가 반복되는 지점(자주 깨지는 검증, 매번 묻는 질문, 헷갈리는 규칙, 누락되는 단계)을 포착해 제안으로 남긴다.
- **제안만 한다 — 자동으로 코드/규칙/문서를 고치지 않는다.** 채택 여부는 사람이 결정한다.
- 제안은 메모리 피드백 형식(무엇을 / 왜 / 어떻게 적용)과 호환되게 적어, 채택 시 바로 반영할 수 있게 한다.

## 실행 방식

- **가볍게 메인 또는 Fork**에서 돈다. 이번 작업의 산출물·로그를 근거로 삼는다.
- **무거운 읽기는 Sub-agent로 내린다.** 여러 작업의 회고·로그를 가로질러 패턴을 찾을 때는 Sub-agent에게 위임하고 메인은 **제안 결론과 경로만** 회수한다. 로그 전문을 메인 컨텍스트에 올리지 않는다.
- **도구 선택은 값어치 기준**: 단일 작업 회고는 메인 혼자로 충분하다. 여러 작업·여러 레포에 걸친 패턴 분석일 때만 Sub-agent로 분담한다.
- **울트라코드(워크플로우) — 항상 작동**: 켜지면 패턴 탐색을 관점별(검증/규칙/흐름/문서) Sub-agent로 병렬화하고 제안의 실효성을 한 번 반박검증한다. 꺼지면 메인이 가볍게 한 바퀴 돈다. 어느 쪽이든 산출물은 동일하다.
- ★자동수정 금지: 이 명령은 변경을 적용하지 않는다. 실제 문서 갱신은 도메인의 `shared/update-docs`가, 코드 변경은 feature/maintenance 트랙이 수행한다. evolve는 **"무엇을 바꾸자"는 제안서만** 만든다.
- 완료기준·증거·리뷰 렌즈는 evolve가 정하지 않는다 — 제안 대상이 속한 플러그인(스택)의 기준을 인용만 한다 (BE/CM = 테스트·lint·build / FE = 두 모드([디자인] 시각·UX·반응형·접근성+Claude 검증, [바인딩] 계약 일치·상태 처리·mock 잔재) / CHAT = +계약 검증·dual review gate).

## 절차

### [V1] 마찰 수집 (메인)

1. 식별자를 resolve하고 이번 작업 산출물을 연결한다: `.harness/artifacts/{track}/{identifier}/`.
2. 이번 작업에서 겪은 마찰을 모은다:
   - evaluate/review에서 반복적으로 걸린 항목
   - 사용자가 같은 질문을 다시 한 지점
   - 우회·임시방편으로 넘어간 부분
   - 규칙/문서(`.harness/docs/*.yaml`)가 모호하거나 최신이 아니어서 헷갈린 부분

### [V2] 패턴 분류 (메인 / Sub-agent)

1. 마찰을 분류한다: `rule`(규칙·컨벤션) / `doc`(문서·진실의 원천) / `command`(하네스 명령·흐름) / `test`(검증 체계) / `process`(절차).
2. **반복성 판정**: 일회성인지, 같은 부류가 과거에도 있었는지. 여러 작업에 걸친 확인이 필요하면 Sub-agent에게 위임하고 결론만 회수한다.
3. 반복되는 것만 제안 후보로 남긴다 (일회성은 기록만).

### [V3] 제안 작성 (메인)

1. 각 후보를 **메모리 피드백 형식**(무엇을 / 왜(반복 근거) / 어떻게 적용)으로 적는다.
2. 적용 대상을 가리킨다: `.harness/docs/*.yaml`(규칙·ADR·컨벤션) / 특정 플러그인 명령 / `CLAUDE.md` / 테스트 체계.
3. **자동 적용하지 않는다.** 우선순위(높음/보통/낮음)만 매긴다.
4. 제안서 `evolve-suggestions.md`를 저장하고, 메인엔 제안 개수와 경로만 남긴다.

## 산출물: evolve-suggestions.md

```markdown
# 개선 제안 (evolve)

## 대상 작업
- 트랙/식별자: {track}/{identifier}
- 한 줄 요약: ...

## 제안 (채택은 사람이 결정 — 자동 적용 안 함)

### [높음] {제목}
- 무엇을: ...
- 왜(반복 근거): {이번/과거에 어떻게 반복됐는가}
- 어떻게 적용: {`.harness/docs/code-convention.yaml`에 ~추가 / `<plugin>` 명령 ~수정 / 테스트 ~보강}
- 적용 주체: shared/update-docs | feature | maintenance | 사람 판단

### [보통] {제목}
- 무엇을: ...
- 왜: ...
- 어떻게 적용: ...

## 일회성 기록 (제안 아님, 참고용)
- ...
```
