# 유사 구현 조사

기존 코드베이스에서 재사용 가능한 패턴과 충돌 가능성을 조사한다.

> **지위**: **opt-in 단독 명령.** `/hb-shared:seed`의 맥락 조사([S2])와 feature:deep의 유사 구현 조사 스텝이 같은 일을 하며, 이 문서는 그 절차의 **스택 중립 canonical 정의**다. 파이프라인 밖에서 이 단계만 따로 돌릴 때 호출한다.

## 실행 방식

이 skill은 Sub-agent가 자동으로 실행한다.

## Sub-agent 프롬프트

```
다음 요구사항을 바탕으로, 기존 코드베이스에서 유사 구현을 조사하라.

조사 항목:
1. 기존 코드에 비슷한 패턴이 이미 있는가? (데이터 구조, API 패턴, 비즈니스 로직, 화면/컴포넌트 구성)
2. 기존 모듈(스택에 따라 모델·서비스·핸들러·컴포넌트 등)을 확장하여 재사용할 수 있는가?
3. 기존 코드와 충돌할 가능성이 있는 부분은? (라우트/URL 충돌, 스키마·필드 충돌, 이벤트/훅 간섭)
4. 기존 테스트 패턴 중 참고할 만한 것은?

용어는 스택을 따른다 — BE=모델/시리얼라이저/뷰, CM/CHAT=핸들러/서비스/레포지토리, FE=컴포넌트/훅/스토어.

[requirements.md]
[module-registry.yaml]
[관련 모듈의 기존 코드]
```
