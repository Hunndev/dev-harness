# 수정 계획 + 회귀 리스크

수정 대상, 범위, 회귀 리스크를 명시하는 계획을 작성한다.

## 실행 방식

이 skill은 Fork에서 실행된다.

## 절차

1. `root-cause.md`, `impact-analysis.md`, `convention-check.md`를 읽는다.
2. 수정 범위를 확정한다:
   - 이번에 수정할 것 (최소 범위 원칙)
   - 이번에 하지 않을 것 (범위 폭주 방지)
3. 각 수정의 회귀 리스크를 평가한다.
4. 초안과 논의점을 사용자에게 제시한다.
5. `fix-plan.md`를 저장한다.

## 범위 제한 원칙

- bug: 근본 원인만 수정. 관련 리팩토링은 별도 이슈로 분리.
- refactor: 동작 변경 금지. 변경이 불가피하면 즉시 중단하고 보고.
- performance: 측정 가능한 개선만. "체감상 빨라진 것 같다"는 불충분.
- dependency: 패키지 업그레이드 후 전체 테스트 통과 필수.

## 산출물: fix-plan.md

```markdown
# 수정 계획

## 수정 대상
| 파일 | 변경 내용 | 이유 |
|------|---------|------|

## 범위 제한
### 이번에 하지 않는 것
- ... — 이유: 별도 이슈로 분리

## 회귀 리스크
| 수정 항목 | 깨질 수 있는 기능 | 리스크 수준 | 대응 |
|----------|----------------|-----------|------|

## 마이그레이션
- 필요 여부: {있음|없음}
- 내용: ...
- 데이터 보정: {있음|없음}

## 수정 순서
1. ...
2. ...
```
