# 영향도 조사 — 병렬 탐색

3개 방향에서 동시에 영향 범위를 탐색한다.
모듈/호출자/데이터 흐름을 동시에 훑어야 누락이 없다.

## 실행 방식

이 skill은 Agent Team으로 실행된다.

## Agent A: 모듈/레이어 방향

```
근본 원인을 기준으로, 수직 방향으로 영향받는 코드를 추적하라.

Model → Serializer → View → URL 순으로:
1. 이 원인이 다른 모델의 FK/관계에 영향을 미치는가?
2. 같은 Serializer를 사용하는 다른 View가 있는가?
3. 다른 앱(module-registry의 다른 모듈)에 전파되는가?

출력: 영향받는 파일:라인 목록 + 영향 내용

[root-cause.md]
[관련 모듈 코드]
[module-registry.yaml]
```

## Agent B: 호출자 방향

```
근본 원인의 코드를 호출하는 모든 경로를 역추적하라.

1. 이 함수/메서드를 호출하는 곳 (import/grep)
2. Signal 연결 (post_save, pre_save 등)
3. Celery task 호출 여부
4. 테스트 커버리지 (이 코드를 커버하는 테스트 수)

출력: 호출 경로 목록 + 각 경로의 영향도

[root-cause.md]
[관련 모듈 코드]
```

## Agent C: 데이터 흐름 방향

```
근본 원인이 데이터 무결성에 미치는 영향을 분석하라.

1. DB 데이터 일관성 영향
2. 이미 잘못된 데이터가 쌓여 있을 가능성
3. 외부 서비스(Innopay, Azure Blob)에 잘못된 데이터 전달 가능성
4. 데이터 보정 마이그레이션 필요 여부

출력: 데이터 영향 목록 + 보정 필요 여부

[root-cause.md]
[관련 모듈 코드]
[architecture.yaml]
```

## 메인: 병합

1. 3개 결과를 통합하여 영향받는 코드/데이터 전체 목록 생성
2. 중복 제거
3. 수정 시 연쇄 영향을 우선순위별로 정리
